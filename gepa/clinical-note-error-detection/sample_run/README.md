# Sample run — Gemini 2.5 Flash Lite × 3.1 Pro reflector

This directory holds a curated snapshot of a single GEPA optimisation run on
the MEDEC validation set. It exists so a reader can see what `detect_gepa.py`
actually produces — the optimised prompt, the headline metrics, and the
Pareto-front evolution plot — without spending API credits to reproduce it.

> **This is a small smoke run, not a benchmark.** The validation slice is
> 25 examples; the full MEDEC-MS validation set is much larger. The numbers
> below illustrate the *shape* of GEPA's improvement on this task, not the
> ceiling of what the method or these models can achieve. For the headline
> benchmark numbers, see the
> [paper](https://arxiv.org/abs/2602.22483) and the
> [original repo](https://github.com/CraigMyles/clinical-note-error-detection).

## Run configuration

| Setting | Value |
| --- | --- |
| Task model | `gemini/gemini-2.5-flash-lite` |
| Reflector model | `gemini/gemini-3.1-pro-preview` |
| Validation slice | 25 examples (12 with errors, 13 correct) |
| GEPA train pool | 100 examples |
| GEPA pareto pool | 25 examples |
| GEPA budget (`auto`) | `light` (480 max metric calls; 498 actually used) |
| Repetitions | 1 |
| Seed | 42 |
| Date | 2026-05-02 |

Full config (including LM kwargs and per-run metrics) is in
[`summary.json`](./summary.json).

## Headline result

| Metric on val (n=25) | Baseline | Optimised | Change |
| --- | --- | --- | --- |
| Error-flag accuracy | 0.520 | 0.600 | +0.080 |
| Balanced accuracy | 0.538 | 0.609 | +0.071 |
| Specificity (correctly saying "CORRECT") | 0.077 | 0.385 | +0.308 |
| False-positive rate | 0.923 | 0.615 | -0.308 |
| MCC | 0.196 | 0.243 | +0.047 |
| Recall (catching real errors) | 1.000 | 0.833 | -0.167 |
| Precision | 0.500 | 0.556 | +0.056 |
| F1 | 0.667 | 0.667 | 0.000 |

The qualitative story this small slice tells: the *baseline* prompt almost
never says "CORRECT" — it flags everything as an error (recall 1.0, FPR 0.92).
GEPA's reflector noticed this and rewrote the prompt to add domain-specific
heuristics for when *not* to flag, raising specificity from 8% to 39% while
giving up only two true positives. F1 stays flat because precision and recall
trade off, but balanced accuracy and MCC both improve, which is what you
actually want on a class-balanced screening task.

## Files

| File | What it is |
| --- | --- |
| [`program.json`](./program.json) | The optimised DSPy program. The `detect.signature.instructions` field is the prompt GEPA evolved; you can paste it into any LLM to reproduce the optimised behaviour outside DSPy. |
| [`summary.json`](./summary.json) | One-stop run manifest: model config, dataset paths, seed, budget, baseline + per-rep optimised metrics, and pointers to the artefacts in the original run dir. |
| [`baseline_val_metrics_overall.json`](./baseline_val_metrics_overall.json) | Overall val metrics for the *unoptimised* prompt (the one taken straight from the paper). |
| [`optimised_val_metrics_overall.json`](./optimised_val_metrics_overall.json) | Overall val metrics for GEPA's best candidate. |
| [`baseline_val_confusion.png`](./baseline_val_confusion.png) | Confusion matrix, baseline. |
| [`optimised_val_confusion.png`](./optimised_val_confusion.png) | Confusion matrix, GEPA-optimised. |
| [`pareto_evolution_score.png`](./pareto_evolution_score.png) | Generated by `src/plot_pareto_front.py`. Shows how the Pareto-front aggregate score evolves as GEPA discovers new candidate prompts; the secondary x-axis shows cumulative metric calls at each discovery point. |

What is **not** here, and why:

- `gepa_log/` (per-iteration JSONs and `gepa_state.bin`) — voluminous and
  opaque; regenerable by re-running.
- `*_predictions.csv`, `*_metrics_per_example.csv` — instance-level dumps;
  useful for ad-hoc analysis but not for explaining the run.
- `*.pdf` versions of the confusion matrices — PNGs are sufficient.

## Reproducing this run

```bash
cd gepa/clinical-note-error-detection
# Set GOOGLE_API_KEY in src/.env or your shell.
bash run.sh
```

The exact CLI invocation is recorded in `summary.json` as the combination of
`preset`, `reflector_preset`, `seed`, `gepa_train_size`, `gepa_pareto_size`,
and `gepa_budget.auto`. Numbers will not match bit-exactly because Gemini API
responses are non-deterministic at temperature > 0 (and `seed` is not
supported by the Gemini provider in LiteLLM), but the *shape* of the
improvement should be reproducible.
